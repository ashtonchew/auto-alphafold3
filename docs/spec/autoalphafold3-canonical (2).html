§ 0 Background — why this problem, why now

Protein folding is one of the cleanest scientific ML stories a hackathon can show on stage: an amino-acid sequence goes in, a three-dimensional backbone comes out, and the audience can see whether the predicted shape is plausible. It is high-impact because structure drives biology: proteins bind, catalyze, signal, misfold, and fail according to geometry.

But the target is not Google DeepMind's real AlphaFold3. Full AlphaFold3 is a large, production-grade biomolecular-complex model. It predicts joint structures for proteins, nucleic acids, small molecules, ions, and modified residues. Its official repository is an inference pipeline with parameter access and use constraints, not a hackathon-scale training substrate. Claiming that we improve, beat, reproduce, or train full AlphaFold3 would be false framing.

The right target is an AlphaFold3-lite research sandbox: small enough to read, train, profile, and patch; real enough that architectural and training changes mean something. That is why the implementation base is ogchen/nanofold. NanoFold is based on the AlphaFold2 and AlphaFold3 papers, is designed to train on a single mid-tier GPU, restricts the problem to monomer protein chains, predicts backbone atoms rather than full biomolecular complexes, and includes a real preprocessing stack for mmCIF structures, MSAs, templates, and Arrow IPC feature dumps.

Core hypothesis

GPT-5.5 can act as an autoresearcher for protein folding, discovering AlphaFold3-lite design improvements that survive falsification. auto-AlphaFold3 wraps a NanoFold-style AlphaFold3-lite implementation in an autoresearch loop: GPT-5.5 pre-registers a falsifiable prediction, makes one architecture, training, loss, sampler, or optimization change, and submits a fixed-budget experiment; Modal runs it through a hardened control plane; the locked evaluator returns validation C-alpha lDDT and a geometry diagnostic report. The unit of progress is not a metric bump — it is a causally-confirmed mechanism. A candidate that improves the score must then survive an automatic falsification gate before it counts as a discovery.

We are using ogchen/nanofold because it is already an AlphaFold3-lite research sandbox, not because it matches real AlphaFold3. The project makes one honest claim: we run autoresearch over a small protein-folding model, under a locked benchmark, and we keep only the design improvements that survive falsification.

0.1  The canonical twist: Fold Cartographer and the Falsification Gate

The differentiated idea is not "Parameter Golf, but for proteins." Parameter Golf taught the compute pattern: smoke small, fan out exploration, drop to one sandbox for debugging, spike validation, and scale to zero when idle. auto-AlphaFold3 adds the layer that turns a metric-chasing loop into a research instrument: every claimed discovery has to survive falsification. Two components carry the thesis.

Fold Cartographer is the diagnostic substrate — the map. The primary score is one number, best validation C-alpha lDDT, but the locked scorer also emits an aggregated geometry diagnostic report. Rather than slicing a tiny validation set into a noisy length × MSA-depth × region cross-product, it reports four orthogonal, literature-grounded failure axes: local geometry, long-range topology, distogram-versus-coordinate decoupling, and stability/compute. This mirrors how the field already reasons about structure confidence — AlphaFold's own pLDDT (local geometry) versus PAE (confidence in the relative position of residue pairs, i.e. long-range topology), and CASP's emphasis on long-range contacts. Because the axes are orthogonal, each one gives the agent a directional claim it can commit to in advance.

The Falsification Gate is what makes those claims mean something. Before a trial, the agent pre-registers a falsifiable prediction: the single change it believes is causal, the one diagnostic axis it predicts will move, the direction, and an honest magnitude band — written down before the run. When a candidate clears the keep threshold, the orchestrator automatically runs a control wave that the agent does not author: a knock-out (the same patch with the credited component removed), a placebo (an unrelated change of matched magnitude), a predicted-axis check (did the axis the agent named actually move as predicted?), and a seed rerun. Only candidates that survive all four become confirmed discoveries.

The integrity property

The agent supplies the claim and a pointer to the component it credits; the orchestrator constructs and runs the controls. Just as the agent cannot edit the scorer, it cannot design its own knock-out or pick its own placebo. This is the line between an agent that grades its own homework and a real control loop — and it is what lets the project say, on stage, that a discovery is a mechanism rather than a lucky checkpoint with a story attached.

Confirmed mechanisms flow into the Discovery Ledger: a provenance-tracked store the agent reads each round, where every entry carries the axis it moves, the falsification verdict and its numbers, the git SHA, the scorer and feature hashes, and a one-line reusable design rule. The trajectory the project presents is this ledger — confirmed mechanisms and the candidates that looked good but were killed at the gate — not a single best checkpoint.

This is folding-specific and performance-relevant. A language-model compression run gets a scalar; you cannot falsify a mechanism story against a single number. A folding run gets a 3D object decomposed into orthogonal geometry axes, so every hypothesis makes a directional prediction you can knock out: long-range contacts missing, local backbone unstable, the pair representation learning contacts while the coordinate path fails, low-MSA examples underperforming, the diffusion sampler too noisy, recycling underused. Fold Cartographer turns those failures into falsifiable predictions; the gate decides which survive.

0.2  Why Modal is still central

Modal is the elastic GPU lab, but the autoresearch agent does not design Modal infrastructure during the event. The compute architecture is predesigned from the Parameter Golf/autoresearch pattern: deploy the experiment app once, call many fixed-budget jobs through a typed runner, write per-trial artifacts to durable Volumes, use one debug sandbox when needed, and scale down while GPT-5.5 thinks. The folding agent spends its reasoning budget on protein-folding hypotheses, not cloud architecture.

The deeper claim is sharper than "we used Modal for GPUs," and sharper than raw parallelism. The Falsification Gate gives parallelism a scientific job: each confirmed discovery is a small parallel control wave — knock-out, placebo, and seeds fired together and collapsed to zero when the verdict is in. We parallelize to run controls, not just to go fast. That is a workload Modal is built for and that a single workstation cannot serve without either idling expensive reservations or serializing the science.

§ 1 Autoresearch from first principles

Autoresearch is not "ask an LLM for ideas." It is a controlled loop, and in auto-AlphaFold3 the loop has a second stage that most autoresearch loops skip: a candidate that improves the score is provisional until it survives falsification.

  1. The agent reads the current best, the latest diagnostics, the Discovery Ledger, and the recent trajectory.
  2. It pre-registers a falsifiable prediction before editing anything: the component it credits, the one axis it expects to move, the direction, and a magnitude band.
  3. It makes one coherent model/training/sampler change.
  4. It submits a typed fixed-budget experiment to the orchestrator.
  5. The orchestrator runs preflight gates and spawns a Modal worker.
  6. The locked scorer returns validation C-alpha lDDT and the geometry diagnostic report.
  7. If the candidate clears the keep threshold, it is a provisional KEEP — not a discovery yet.
  8. The orchestrator constructs and runs the falsification wave: knock-out, placebo, predicted-axis check, seed rerun.
  9. The verdict is computed — confirmed, or killed with a named reason.
  10. A confirmed mechanism is committed and written to the Discovery Ledger with full provenance; a kill is logged honestly with its reason.
  11. The agent updates its picture from the verdict, not just the score.
  12. The local orchestrator writes the canonical ledger entry.

For auto-AlphaFold3, the scalar is validation C-alpha lDDT. The editable surface is NanoFold-style architecture and training code. The fixed budget is a small training or inference-only sampler run on preprocessed Arrow features. The compute substrate is Modal. The locked contract is the validation split, scorer, raw labels, result parser, baseline ledger — and the falsification controls, which the orchestrator owns.

Autoresearch contract

The agent may edit the model/training/sampler search surface, but not the benchmark contract, the Modal control plane, or the falsification controls. If it can change the scorer, split, validation labels, preprocessing outputs, result parser, infrastructure guardrails, or the knock-out/placebo/threshold logic, the project stops being research and becomes evaluator hacking. The exact editable and locked file lists are canonical in §4.

The research claim is the trajectory, not a single lucky checkpoint — and the falsification gate is what earns that claim. A good run reads like this: "The agent saw long-range contact precision improve while C-alpha lDDT stayed flat; it inferred the pair representation was learning distances the coordinate path was not using; it pre-registered that ramping the geometry loss would close the distogram-versus-coordinate gap and lift lDDT. The score improved. The gate's knock-out removed the geometry ramp and the gain vanished, so the gain was attributable to the credited change; a placebo in the optimizer family produced no gain, so it was not generic regularization; the predicted gap closed as predicted; the gain beat seed noise. It entered the Discovery Ledger. The next candidate, a flashy attention tweak, also raised the score — but its placebo reproduced most of the gain, so it was killed as noise." That is an autoresearch story, not a grid search, and the kills are part of it.

§ 2 Why auto-AlphaFold3 fits the shape

Folding-model autoresearch is heavier than probe autoresearch, but it has a stronger demo and a richer research surface. It fits the shape for seven reasons.

1. AlphaFold3-lite gives meaningful architecture surface

NanoFold is not a black-box API. It exposes input embedding, MSA processing, template features, Pairformer-style reasoning, recycling, diffusion-style coordinate generation, distogram loss, diffusion/coordinate loss, training, dataset streaming, checkpointing, gradient checkpointing, and torch.compile. GPT-5.5 can make scientifically legible changes.

2. The metric is a scalar judges understand

Best validation C-alpha lDDT asks: how close is the predicted protein backbone geometry to the true structure, using alpha carbon positions? Higher is better. Loss curves, contact precision, runtime, memory, parameter count, NaNs, and OOMs are diagnostics. The scoreboard is one number.

3. Folding gives richer feedback than Parameter Golf

Parameter Golf has scalar compression loss. Folding gives geometry. The same C-alpha lDDT miss can be local loop instability, bad long-range packing, low-MSA failure, pair-contact failure, coordinate-head failure, or diffusion-sampler failure. Fold Cartographer gives the agent a domain-specific map of what went wrong.

4. Training trials are feasible with tiny configs and Modal

A full folding model is expensive. A NanoFold-style monomer model on a small fixed train subset is manageable. The event plan uses tiny crop sizes, reduced MSA depth, reduced trunk depth for smoke, short training windows, single-seed exploration, inference-only sampler trials, and final multi-seed validation only for confirmed candidates.

5. Diffusion sampler golf is a cheap inner loop

Because NanoFold has a diffusion module with inference-time schedule knobs, the agent can run inference-only sampler experiments on a fixed checkpoint. Sampler trials search diffusion steps, noise schedule, step scale, sample count, and label-free sample selection. They are far cheaper than retraining and directly tied to AlphaFold3-lite design — and, crucially, the same cheap inference path runs the gate's knock-outs and placebos for sampler-family mechanisms.

6. Structure prediction creates visual artifacts

Protein folding produces shapes. The demo can show a baseline predicted backbone, the true backbone, the agent-improved prediction, the geometry error report, and the Discovery Ledger that separates confirmed mechanisms from killed candidates. Even a modest numerical improvement is easier to sell when paired with a visible overlay and a mechanism that has been falsification-tested.

7. Folding gives falsifiable feedback, not just a scalar

This is what the gate runs on. A scalar compression loss can rise or fall, but it cannot be falsified mechanistically — there is no orthogonal axis to knock out. A folding run decomposes into four orthogonal axes, so a hypothesis can name the axis it expects to move and the direction, and the orchestrator can test whether the named axis actually moved, whether removing the credited component erases the gain, and whether an unrelated change reproduces it. Folding is the rare hackathon-scale task where "the agent did real science" can be demonstrated rather than asserted.

SimplexFold as inspiration, not base

SimplexFold is useful because it shows the ambition bar: a bold folding architecture hypothesis, explicit higher-order geometric states, benchmark discipline, and trajectory-ready artifacts. auto-AlphaFold3 does not use SimplexFold as the implementation base. It uses NanoFold / AlphaFold3-lite as its base.

§ 3 The science, plainly

The biology and the method can be explained without drowning judges in jargon. Use these definitions consistently.

Protein sequence input
A protein is a chain of amino acids. The sequence is the one-dimensional string. Folding predicts how that string sits in three-dimensional space.
Protein structure output
The 3D arrangement of the atoms in the protein. In this hackathon, we care mainly about monomer backbone geometry, not full complexes with ligands, DNA, RNA, or multiple protein chains.
C-alpha backbone simplified geometry
Each amino acid has an alpha carbon. Connecting those alpha carbon positions gives a clean trace of the protein backbone. It is less detailed than all-atom structure but excellent for fast scoring and visualization.
lDDT local distance difference test
A structure-quality score based on whether local distances in the predicted structure match local distances in the true structure. C-alpha lDDT applies this idea to alpha carbon positions. Plain English: how close the predicted protein backbone geometry is to the true structure, using the alpha carbon positions.
MSA multiple sequence alignment
A stack of related protein sequences from evolution. Correlated mutations can reveal which residues touch in 3D. Folding models use MSAs as evolutionary hints.
Templates known related structures
Existing protein structures that resemble the target sequence. Templates can help when a related fold has already been solved, but this hackathon benchmark disables them with max_templates=0 so every trial folds from the same no-template input contract.
AlphaFold2 vs. AlphaFold3 high level
AlphaFold2 popularized modern deep-learning protein structure prediction with MSA/template features, pair representations, Evoformer-style reasoning, recycling, and a structure module. AlphaFold3 broadened the target to biomolecular complexes and moved toward Pairformer-style reasoning plus diffusion-based coordinate generation. auto-AlphaFold3 borrows the AlphaFold3-lite flavor; it does not reproduce full AlphaFold3.
NanoFold / AlphaFold3-lite our sandbox
A small, hackable protein-folding implementation inspired by AlphaFold2 and AlphaFold3. It is monomer-focused, trainable on modest GPUs, and appropriate for agent-driven architecture/training/sampler search.
Fold Cartographer the diagnostic substrate
A locked diagnostic layer that converts folding failures into aggregated geometry signatures along four orthogonal axes: local geometry, long-range topology, distogram-versus-coordinate decoupling, and stability/compute. The local-versus-long-range split mirrors AlphaFold's pLDDT-versus-PAE confidence pair. The orthogonality is what lets a hypothesis name a single axis to be tested against.
Falsifiable prediction pre-registration
Before a run, the agent writes down the component it believes is causal, the one axis it expects to move, the direction of the move, and an honest magnitude band. Writing it before the result is what makes the later check a test rather than a story.
Knock-out attribution control
The same candidate patch with the credited component removed. If the score gain survives the knock-out, the gain was not caused by the component the agent credited — the mechanism story is wrong even if the improvement is real.
Placebo noise control
An unrelated change of similar magnitude in a different move family. If the placebo reproduces most of the gain, the "improvement" is likely generic regularization or validation-set noise, not a specific mechanism.
Attribution fraction how much is real
The share of the score gain that disappears when the credited component is knocked out. A confirmed mechanism requires most of the gain to be attributable to the component the agent named.
Falsification Gate our twist
The orchestrator-run control wave — knock-out, placebo, predicted-axis check, seed rerun — that a provisional KEEP must survive to count as a discovery. The agent supplies the claim; it does not author the controls.
Confirmed mechanism what the agent discovers
A change to architecture, loss, training curriculum, optimizer, recycling, diffusion/noising, sampler, feature handling, or runtime that improves locked validation C-alpha lDDT, where the gain is attributable to the credited component, is not reproduced by a placebo, moves the predicted axis as predicted, and beats seed noise — all without changing the evaluator, the validation split, or the controls.
Discovery Ledger cumulative record
The provenance-tracked store of confirmed mechanisms the agent reads each round. Each entry carries the axis it moves, the verdict numbers, git SHA, scorer and feature hashes, and a one-line reusable design rule.

§ 4 Methodology & search space

The agent should not be restricted to a single config file. That would make sense for a frozen-probe plan; it is too weak for folding architecture research. The agent may edit a broad but designated search surface: selected NanoFold model/training modules, an autoalphafold3/ patch area, configs, loss definitions, sampler wrappers, and training schedules. It may not edit scoring, split, preprocessing output, result parser, baseline ledger, the Modal control plane, or the falsification controls.

Search families

  • Architecture width/depth. Pair representation width, single representation width, number of trunk blocks, transition widths, head dimensions.
  • Pairformer / attention variants. Pair attention bias, triangle update simplifications, dropout placement, gated residuals, norm order, attention head count.
  • Diffusion / noising schedule. Training noise schedule, diffusion steps, loss weighting by noise level, train/sampling mismatch.
  • Diffusion sampler golf. Inference-only search over sampler steps, gamma_0, gamma_min, noise_scale, step_scale, s_max, s_min, p, number of stochastic samples, and label-free sample-selection rules.
  • Recycling. Number of recycles, random recycle range during training, recycle schedule, and early-run reductions for speed.
  • Auxiliary losses. Distogram/contact objectives, confidence heads, pLDDT-like diagnostics, alignment losses, and loss ramps.
  • Geometry losses. Coordinate loss, local-distance loss, bond/angle regularizers, backbone-only losses, clash proxies when cheap.
  • Distogram/contact losses. Distance-bin loss, contact precision diagnostics, harder contact weighting, long-range contact weighting.
  • Curriculum. Crop length, MSA depth, short-to-long sequence schedule.
  • Optimizer/scheduler. AdamW parameters, weight decay, warmup, cosine decay, gradient clipping, accumulation, batch size.
  • Memory/runtime optimization. bfloat16, gradient checkpointing, torch.compile toggles, batch sizing, model width reductions that permit more steps.
  • Feature handling. MSA dropout, feature normalization, empty-template placeholder behavior, MSA subsampling.

Fold Cartographer diagnostic axes

Before each hypothesis, the agent reads the latest error_report.json and registers its prediction against exactly one of four orthogonal axes. Each is one robust aggregate computable on a tiny validation set — not a fine-grained bucket cross-product — and each routes to a distinct move-family cluster and defines what the gate's predicted-axis check will measure.

  • local_geometry_weak (pLDDT-like; mean local lDDT, short-range contact precision) → geometry loss, local backbone regularizers, termini/loop augmentation.
  • long_range_topology_weak (PAE-like; long-range contact precision, sequence separation ≥ 24) → Pairformer/triangle updates, distogram weighting, crop curriculum.
  • distogram_good_lddt_flat (pair head learns contacts but the 3D path does not follow) → coordinate loss, recycling, diffusion sampler.
  • stability_compute (NaN/OOM rate, runtime, peak memory) → width/depth, gradient clipping, checkpointing, torch.compile.

Sequence length and MSA depth are reported as two or three coarse buckets for context only; the agent does not register predictions on them, because per-bucket estimates on a small validation set are too noisy to falsify.

The falsification contract

Every hypothesis must declare, before the run, a structured prediction: the single causal_component it credits, the predicted_axis it expects to move, the predicted_direction, and an honest expected_lddt_delta_band. The agent authors the prediction. The orchestrator authors and runs the controls. A provisional KEEP that does not survive them is not a discovery.

    gain_placebo: float                 # lddt(parent + unrelated change) - lddt(parent)
    attributable_fraction: float        # (gain_full - gain_knockout) / gain_full
    axis_delta_observed: float
    axis_prediction_held: bool
    seed_mean: float
    seed_std: float
    verdict: Literal[
        "CONFIRMED", "KNOCKOUT_SURVIVES", "PLACEBO_KILL",
        "AXIS_MISS", "SEED_FRAGILE",
    ]

@dataclass

    mechanism: str                       # one-line confirmed claim
    axis_moved: Axis
    design_rule: str                     # reusable rule the agent reads next round
    provenance: dict                     # git_sha, scorer_version, features_hash, verdict numbers

@dataclass
class AutoFoldResult:
    trial_id: str
    status: Literal["KEEP", "DISCARD", "FAIL", "INFRA_FAIL"]   # KEEP is PROVISIONAL
    discovery: Literal["CONFIRMED", "UNCONFIRMED", "KILLED"]
    primary: dict[str, float]
    falsification: FalsificationResult | None
    git_commit: str | None
    postmortem: str

Decision rule — two stages

Stage one (threshold). A trial is a provisional KEEP if it improves best validation C-alpha lDDT over the current best by the configured threshold and does not create unacceptable NaN/OOM/runtime behavior. It is DISCARD if it fails the metric, violates budget, or only improves by noise. It is FAIL if the training path breaks; INFRA_FAIL if the Modal worker or artifact path fails independent of the patch.

Stage two (the gate). Every provisional KEEP runs the falsification wave, and the verdict decides whether it is a discovery. The controls are computed against the same locked scorer and public split; no new labels are introduced and the hidden split is never touched here.

  • CONFIRMEDEnters the Discovery Ledger. The gain is attributable to the credited component (attributable_fraction ≥ tau), the placebo did not reproduce it (gain_placebo < rho · gain_full), the predicted axis moved as predicted, and the gain beats seed noise (gain_full > k · seed_std).
  • KNOCKOUT_SURVIVESMisattributed. Removing the credited component kept the gain. The improvement may be real, but the agent's mechanism story is wrong; it must re-hypothesize before claiming anything.
  • PLACEBO_KILLLikely noise or generic regularization. An unrelated change of similar size reproduced most of the gain, so the specific mechanism is not supported.
  • AXIS_MISSReal but unexplained. lDDT rose but the predicted axis did not move in the predicted direction. Logged honestly as an unexplained improvement, not a confirmed mechanism.
  • SEED_FRAGILEWithin seed noise. The gain does not survive a seed rerun on a tiny validation set; discarded.

During broad search, single-seed exploration is fine before the gate fires; the gate's seed rerun is where seed stability is judged. During final validation, confirmed discoveries get additional seeds on the fixed public split, plus optional hidden validation if staged before the event. The hidden split is final-only; it is never used for diagnostics, threshold decisions, or gate controls.

§ 5 Agent stack & tooling

The autoresearch agent is not told to design the compute plan. That is the creator's job, and it is already encoded in a hardened Modal control plane. The folding agent focuses on NanoFold / AlphaFold3-lite research hypotheses and the falsifiable predictions attached to them.

Corrected Modal contract

The Modal compute architecture is creator-designed. The agent submits AutoFoldTrial JSON objects through the local orchestrator. It may not call modal run directly, spawn arbitrary sandboxes, modify Volumes, change GPU types, raise max_containers, edit autoalphafold3/modal_app.py, or author the falsification controls during search.

5.1  Control-plane / worker split

The platform has three layers:

  • GPT-5.5 agent. Reads repo, configs, logs, diagnostics, and the Discovery Ledger. Writes one hypothesis, one pre-registered prediction, one patch, one AutoFoldTrial.
  • Local orchestrator. Validates patch scope, checks forbidden files, enforces budget/resource tier, runs preflight, submits to the deployed Modal function, polls results, runs the falsification gate, computes the verdict, and writes the canonical ledger and Discovery Ledger.
  • Modal workers. Trial/sampler workers load preprocessed Arrow features, apply the patch in an isolated workspace, and write predictions/checkpoints under trial-scoped /runs paths. A separate scorer-only worker mounts locked labels/manifests, computes metrics, and returns canonical diagnostics.

This separation keeps the scientific search broad while narrowing the operational surface. The agent makes real architecture/training/sampler changes and authors falsifiable predictions, but it cannot mutate the infrastructure that enforces cost, locking, scoring, or the controls that decide what counts as a discovery.

5.2  Deploy once, call many

The Modal app is deployed once before the search loop. Trials are launched through modal.Function.from_name(...).spawn(...) or .remote(...); the gate's control wave is launched through .starmap(...) over (variant, seed) tuples. Do not issue many concurrent modal run processes. This avoids app teardown conflicts, keeps images and Volume mounts consistent, and makes job polling deterministic.

    run_one_batch_loss_check(trial)
    run_scorer_dry_run_schema_check()

def submit_trial(trial: dict) -> str:
    preflight(trial)
    fn = modal.Function.from_name(APP_NAME, "run_trial")

def poll_call(call_id: str, timeout_s: int = 3600) -> dict:
    call = modal.FunctionCall.from_id(call_id)
    start = time.time()
    while time.time() - start < timeout_s:
        try:
            return call.get(timeout=0)
        except TimeoutError:
            time.sleep(10)
    return {"status": "INFRA_FAIL", "reason": "timeout", "call_id": call_id}

    LEDGER.parent.mkdir(parents=True, exist_ok=True)

5.9  Falsification gate execution

When a trial returns a provisional KEEP, the orchestrator builds the controls and fires them as a parallel wave. The knock-out is constructed mechanically by reverting the declared causal_component from the candidate patch; the placebo is drawn from an unrelated move family at matched magnitude; the seed rerun repeats the candidate across n_seeds. For sampler-family mechanisms the controls are inference-only on the frozen checkpoint, so they are seconds long and ride the cheap path. Exceptions are treated as first-class results so a FAILed control becomes a clean kill rather than a crashed wave.

autoalphafold3/falsification.py — wave + verdict (locked)

import modal

def run_gate(candidate, parent_lddt, plan) -> "FalsificationResult":
    knockout = build_knockout(candidate, plan.knockout_patch)      # orchestrator-built
    placebo  = build_placebo(candidate.parent_commit, plan.placebo_family)
    seeds    = [(candidate, s) for s in range(plan.n_seeds)]

    run = modal.Function.from_name("autoalphafold3", "run_trial")
    # one parallel wave: knock-out, placebo, and N seed reruns fired together.
    # return_exceptions keeps a broken control from killing the verdict;
    # wrap_returned_exceptions=False returns the real exception type (modal>=1.4).
    variants = [(knockout, 0), (placebo, 0), *seeds]
    artifacts = list(
        run.starmap(variants, return_exceptions=True, wrap_returned_exceptions=False)
    )
    metrics = score_all(artifacts)        # scorer-only path; same locked split/labels

    g_full = candidate.lddt - parent_lddt
    g_ko   = metrics["knockout"].lddt - parent_lddt
    g_pl   = metrics["placebo"].lddt - parent_lddt
    attributable = (g_full - g_ko) / g_full if g_full else 0.0
    axis_held = axis_moved_as_predicted(metrics, candidate.prediction)
    seed_mean, seed_std = summarize_seeds(metrics["seeds"])

    if g_full <= plan.k_seed * seed_std:        verdict = "SEED_FRAGILE"
    elif g_pl >= plan.rho_placebo * g_full:      verdict = "PLACEBO_KILL"
    elif attributable < plan.tau_attribution:    verdict = "KNOCKOUT_SURVIVES"
    elif not axis_held:                          verdict = "AXIS_MISS"
    else:                                        verdict = "CONFIRMED"
    return build_result(locals())

The gate fires only on provisional KEEPs, which are rare, and most controls are inference-only, so the marginal cost is small. The full ceremony is also a scope dial: under time pressure, run it on the single best mechanism and show one clean kill alongside it.

Data is the make-or-break constraint. Full folding preprocessing is too slow and too failure-prone to discover on event day.

Data rule

Preprocess before event. During the event, never run full MSA/database preprocessing except tiny smoke checks. The hackathon starts from preprocessed Arrow IPC features, a tiny train subset, a fixed public validation split, and a working NanoFold default/tiny baseline.

6.1  NanoFold data path

NanoFold consumes PDB/mmCIF structural inputs, searches sequence databases for MSAs, can search template databases, and writes Arrow IPC feature dumps. The relevant preprocessing for this hackathon includes mmCIF parsing from PDB, small BFD / Uniclust30-style sequence search for MSAs, empty template placeholders, and Arrow IPC feature output. This is real protein-folding data work, not something to debug after kickoff.

No-template benchmark

Pin max_templates=0 for all official runs. A template is a previously solved structure of a similar protein. Allowing templates changes the task: the model can lean on an external solved-shape hint instead of folding from sequence, MSA/evolutionary signal, and learned geometry alone. It also makes setup heavier because a template database must be provisioned, searched, versioned, and locked. The simpler and cleaner hackathon contract is no-template NanoFold: template tensors may exist as empty schema placeholders, but the event does not require template DB access, template search, template-sensitive diagnostics, or agent control over template retrieval. This reduces leak risk and keeps every baseline and agent trial on the same input contract.

  • Train subset. Fixed small subset of preprocessed monomer chains for fast experiments. Size is chosen by runtime, not ambition.

  • Baseline ledger. Contains NanoFold default/tiny baseline metrics, immutable after T+1.

6.5  Metric, diagnostics, and the gate's data boundary

The primary scalar is best validation C-alpha lDDT. Diagnostics are the four orthogonal Fold Cartographer axes — local geometry (local lDDT, short-range contact precision), long-range topology (long-range contact precision), distogram-versus-coordinate gap, and stability/compute (runtime, memory, parameter count, NaN/OOM rate) — plus coordinate/distogram loss and final seed variance. lDDT by coarse length bucket is reported for context only, never registered on. Diagnostics are emitted by the locked scorer and summarized by Fold Cartographer.

The Falsification Gate consumes the same locked scorer and the same public split as the primary metric. It introduces no new labels and never touches the hidden split; knock-out, placebo, and seed controls are scored exactly as ordinary trials are. The gate is a stronger evidentiary bar, not a backdoor to extra data.

6.6  Default/tiny baseline

The first baseline is the upstream NanoFold default path on the smallest feasible setup. The second is a local auto_tiny config that preserves the training/evaluation path while shrinking crop length, MSA depth, trunk depth, diffusion steps, batch size, and train steps for fast smoke experiments. The public scoreboard compares agent candidates against the best baseline available by T+1.

§ 7 Where Modal comes in

Modal is the elastic GPU lab for autonomous folding experiments. The deeper claim is sharper than "we used Modal for GPUs." Parameter Golf proved Modal can elastically scale how much compute an agent uses; auto-AlphaFold3 proves it can elastically scale what kind, and that the gate gives that elasticity a scientific job. A single folding-research session naturally spans five sharply different execution modes, and a confirmed discovery is itself a small parallel control wave. NanoFold-lite is primarily independent single-GPU training and inference, so this plan uses parallel independent single-GPU jobs by default; multi-GPU is useful only if the training code actually runs distributed execution, which it does not here.

The five-mode + control-wave thesis

Folding autoresearch demands five distinct execution modes in one session: a massive parallel CPU fan-out for data, cheap single-GPU training trials, a few-hundred-job inference-only sampler burst, a one-Sandbox debug drop, and a final validation/render spike — then scale to zero while GPT-5.5 thinks. The Falsification Gate adds a sixth pattern composed from the others: every provisional KEEP fans out a knock-out, a placebo, and N seed reruns as one parallel wave that collapses to zero when the verdict is in. We parallelize to run controls, not just to go fast. The original autoresearch run hit roughly 113 experiments across 238 GPU-hours on a near-uniform language-model task; folding's work is not uniform, and the controls make the non-uniformity the point.

7.1  Execution modes

MODE A
Parallel CPU fan-out — build the benchmark
Function.map · Volume.commit · pre-event · CPU · scale → 0

mmCIF parsing and MSA search over hundreds of monomer chains is embarrassingly parallel, heavy, and one-shot. spawn → FunctionCall.get · gpu="A100-80GB" (pinned) · max_containers 2–6

All official trials run on one pinned A100-80GB Function: 80 GB covers the tiny no-template crops, and a single fixed tier keeps runtime and peak-memory numbers in the stability_compute diagnostic comparable across trials. Training-family knock-outs and placebos in the gate run on this same pinned Function.

Tier: trial. The main loop.

MODE C
Inference-only sampler burst — the signature
Function.map · gpu="A100"/"L40S" · max_containers ~50 · scaledown_window 20s

Because the diffusion sampler runs on a frozen checkpoint, hundreds of sampling configs fire in one sample_once.map(grid) wave, each a few seconds long. The same cheap path runs the gate's knock-outs and placebos for sampler-family mechanisms, and returns label-free sample artifacts rather than committing to the Volume, which avoids concurrent-commit contention.

Tier: sampler. The cheapest, most parallel, most visible Modal moment on stage; it has no analog in Parameter Golf.

MODE D
Interactive debug Sandbox — triggered drop
Sandbox.create(image, gpu) · one at a time · exec inspect

Open one debug Sandbox only on explicit triggers: three consecutive shape errors in the same family, two OOMs after a budget downgrade, NaNs in both baseline and candidate paths, or scorer failure on a previously valid checkpoint. modal.Sandbox.create() provisions an exec-able GPU container with the training image in seconds; inspect the checkpoint, shape, or OOM, then terminate.

Tier: debug. Drops are orchestrator/human-triggered, not agent-discretionary, and serial by design.

MODE E
Validation spike — multi-seed + render, then zero
fan-out candidates × seeds · gpu="H100" · render overlays · scaledown_window 60s

Final-only. At T+7, stop broad search; rerun confirmed discoveries across multiple seeds on the locked public split; use hidden validation only here; render predicted-vs-true backbone overlays for baseline and best candidate. Scale up briefly, validate honestly, scale down.

Tier: final. The only place hidden validation is touched.

After each trial the scorer-only function returns canonical metrics plus an error_report payload; Fold Cartographer summarizes the failure signature, and the agent registers a prediction before the next patch. Provisional KEEPs trigger the gate; metrics, diagnostics, and verdicts return to the orchestrator (Fig. 7), so the loop reasons over confirmed mechanisms, not a single scalar, and the scorer never needs writable access to any trial or control artifact mount.

7.4  How the agent's code edits reach Modal

A deployed Modal function runs the code captured when it was deployed, so the loop needs a disciplined answer to one question: how does a GPT-5.5 patch to NanoFold reach the GPU container — fast, and reproducibly? Separate the slow-changing image from the fast-changing code, then run the loop at two rates: redeploy only on a code change, and spawn / map / starmap freely on config-only changes.

The rule

Pin the image (CUDA base + exact library versions) so heavy layers cache and stay reproducible. Add the editable dirs with copy=False so they mount at container startup instead of baking an image layer — a code patch then re-uploads only the mount, not a layer. Redeploy when the editable-surface git SHA changes; otherwise just submit more jobs against the current deployment. Because the gate reuses run_trial, the knock-out and placebo run the exact code the wave was frozen at — the controls and the candidate can never run different source.

With copy=False, a container reads the editable dirs at startup. The working tree is mutated exactly once per wave — apply patch, commit, deploy-if-changed, launch trial, wait, then run the gate against the same frozen SHA — and the next patch waits until the verdict resolves. Every trial and every control records the exact code, app revision, scorer, and features it ran against, so the research trajectory is reproducible, not just fast.

§ 10 Risk assessment

  • Preprocessing takes too long. MSA preprocessing can consume the whole event.Preprocess before event. Store Arrow IPC features in autoalphafold3-data. During the event, only run tiny smoke checks against prepared paths. Keep templates disabled with max_templates=0.
  • Training loop is too slow for 10 hours. Folding models are heavier than probe heads.Use auto_tiny, short budgets, sampler-only inner loops, and final multi-seed only for confirmed candidates.
  • Agent mutates infrastructure or controls. If the agent edits the Modal app, scoring, budget caps, or the gate, the loop becomes unsafe.Modal app, scorer, and falsification.py are locked. Agent submits typed trials and authors only its own prediction. Orchestrator checks forbidden files before every run.
  • Gate cost. The control wave 2–3× the GPU cost of a provisional KEEP.Fire the gate only past threshold; KEEPs are rare. Prefer inference-only knock-outs/placebos on the frozen checkpoint. Treat the full ceremony as a scope dial — run it on the single best mechanism under time pressure.
  • Gate too strict, kills everything. Aggressive thresholds on a tiny validation set can reject real mechanisms.Calibrate tau, rho, and k_seed on a known-null and known-positive before T+2. An AXIS_MISS is logged as a real-but-unexplained improvement, not discarded.
  • Agent games the gate. A self-served control can be designed to be toothless.The agent supplies only the claim and the credited component. The orchestrator constructs the knock-out and draws the placebo; the agent cannot author or edit controls or thresholds.
  • Default NanoFold random split violates benchmark lock. A ratio split is not a locked evaluator.Use explicit manifests and manifest hashes through locked_dataset.py. Refuse official trials using random split.
  • Loss-only evaluation lacks prediction C-alpha lDDT. NanoFold's training path reports losses; the demo needs structures and lDDT.Implement separate prediction-artifact generation and scorer-only lDDT, separated by the Volume boundary.
  • Agent creates unstable architecture. A broad editable surface can break tensor shapes.Patch through scaffolded patterns; add shape tests, param-count checks, tiny forward pass, and automatic rollback for failure states.
  • NaNs and OOMs dominate the ledger. Diffusion losses and mixed precision can destabilize.Treat NaN/OOM as FAIL. Log honestly, downgrade budget only within rules, and use one debug sandbox under trigger rules.
  • Validation C-alpha lDDT is noisy. Small validation subsets produce false positives.This is exactly what the seed and placebo controls catch. Use a keep threshold above baseline noise; the gate's SEED_FRAGILE and PLACEBO_KILL verdicts reject noise before it reaches the Discovery Ledger.
  • Diagnostics leak labels or become objective hacking. Rich diagnostics can tempt overfitting to public validation.Diagnostics are aggregated, locked, and public-val only. The gate uses the same locked split, no new labels. Hidden validation remains final-only. Primary metric stays best validation C-alpha lDDT.
  • Shared ledger writes collide. Multiple workers appending one file can lose data.Workers write only per-trial directories. The local orchestrator is the single ledger and Discovery Ledger writer.
  • Modal GPU image issues. Torch, CUDA, pyarrow, and NanoFold deps can fail in the container.Build and test the image before event. Pin dependencies. Keep a minimal fallback image that runs tiny baseline and scorer.
  • Unfair "AlphaFold3" overclaim. Judges may infer we modified real AlphaFold3.Repeat the framing everywhere: NanoFold-style AlphaFold3-lite sandbox, not full AlphaFold3.
  • Demo does not improve score. The final candidate may only match baseline or improve modestly.Build the demo around the trajectory, the gate, confirmed mechanisms, instructive kills, Modal jobs, and overlays. A modest confirmed mechanism with a clean kill beside it still lands.

§ 11 The demo

The demo is not a leaderboard screenshot. It is a research story the audience can watch resolve. The score matters, but the thing that makes auto-AlphaFold3 land is visible on stage: the agent proposes a mechanism, the orchestrator puts it on trial, and the verdict — confirmed or killed — is shown live. The artifact has five parts.

1. The trajectory

A single plot of best validation C-alpha lDDT over trials, with each point colored by move family and shaped by outcome: a hollow marker for a provisional KEEP, a filled green marker for a CONFIRMED discovery, and a red cross for a candidate killed at the gate. The line that matters is not the score curve alone — it is the gap between "looked good" and "survived." A run where every provisional KEEP became a discovery would be suspicious; a run with visible kills is a run with a working control loop.

2. The Discovery Ledger, on stage

The centerpiece. A two-column panel: confirmed mechanisms on the left, killed candidates on the right with their verdict reasons. The red column is the moment that sells the integrity property — the agent visibly caught its own false positives. An example of what the audience sees:

Confirmed mechanisms
  • Geometry-loss ramp closes the distogram-to-coordinate gap; +0.041 lDDT, attributable 0.78, axis held, seed-stable.axis: distogram_vs_3d · geometry_loss
  • Long-range contact weighting lifts topology on sep≥24; +0.026 lDDT, attributable 0.63, placebo null.axis: long_range_topology · auxiliary_loss
  • Sampler step-scale + fewer steps holds lDDT at lower cost; +0.018 lDDT, attributable 0.71.axis: distogram_vs_3d · diffusion_sampler_golf
Killed candidates
  • Wider pair attention "improved" lDDT — placebo of equal size reproduced 0.7 of the gain.PLACEBO_KILL
  • Extra recycling step raised the score, but knock-out kept it — credit was misattributed.KNOCKOUT_SURVIVES
  • Dropout tweak lifted lDDT but local-geometry axis never moved.AXIS_MISS · logged unexplained
  • Norm reorder gain vanished on a seed rerun.SEED_FRAGILE

3. The falsification card for the headline discovery

For the single best confirmed mechanism, a card that shows the pre-registered claim, the axis it predicted, and the four control bars side by side — full gain, knock-out gain, placebo gain, and seed mean with its spread. The eye reads the verdict before the numbers: the full bar is tall, the knock-out collapses, the placebo is flat, and the seed band is narrow.

4. The live Modal moment

Two visible spikes. First, the sampler-golf burst (Mode C): a live Function.map over a few hundred frozen-checkpoint sampling configs; the Modal dashboard's container count climbs toward fifty and falls back to zero in about a minute, and the samples render as a conformational ensemble — rigid core tight, flexible loops smeared. Second, the gate: when the headline discovery is put on trial, a small control wave spikes and collapses, and the verdict resolves on screen.

5. The structure overlay

Predicted-versus-true C-alpha backbone overlays for the baseline and the best confirmed candidate, with the geometry error map beside them. The audience sees the shape get closer to truth and the named axis improve in the same frame.

The winning frame

The strongest single screen is the Discovery Ledger with the falsification card open on the headline mechanism and one instructive kill highlighted beside it. It says, without narration: the agent did not just raise a number — it proposed mechanisms, tested them against controls it did not author, kept the ones that survived, and threw out the ones that did not.

§ 14 Starter program.md

This is the operating doc handed to the agent at T+0. It is deliberately strict: the agent's freedom is in the science, not the infrastructure or the controls.

program.md — agent instructions

# auto-AlphaFold3 — autoresearch operating instructions

## Objective
Improve best validation C-alpha lDDT on the locked public split by discovering
AlphaFold3-lite design improvements in the NanoFold-style sandbox. A higher
score is necessary but not sufficient: only mechanisms that SURVIVE THE
FALSIFICATION GATE count as discoveries. Chase confirmed mechanisms, not bumps.

## The loop (one change at a time)
1. Read: current best, latest error_report.json, the Discovery Ledger, recent trials.
2. Pre-register a falsifiable prediction (see format) BEFORE editing anything.
3. Make ONE coherent change in an allowed move family.
4. Submit a typed AutoFoldTrial through the orchestrator. Never call modal directly.
5. Read the returned metrics and diagnostics.
6. If it clears the keep threshold, it is a PROVISIONAL KEEP — not a discovery.
   The orchestrator will run the gate. Wait for the verdict.
7. On CONFIRMED: it is committed and written to the Discovery Ledger. Build on it.
   On any kill: read the reason, update your picture, re-hypothesize. Do not retry
   the same change hoping for a different verdict.
8. Move to the next hypothesis.

## Pre-registration format (required before every run)
prediction:
  causal_component: ""
  predicted_axis:   one of {local_geometry, long_range_topology,
                            distogram_vs_3d, stability_compute}
  predicted_direction: up | down
  expected_lddt_delta_band: [low, high]   # honest; do not pad
Register against EXACTLY ONE axis. If you cannot name a single axis and direction,
you do not yet have a hypothesis — keep reading diagnostics.

## Two-stage decision
Stage 1 (threshold): KEEP if best val C-alpha lDDT improves over current best by
the threshold without unacceptable NaN/OOM/runtime. Else DISCARD / FAIL / INFRA_FAIL.
Stage 2 (the gate): the orchestrator runs knock-out, placebo, axis-check, and seed
rerun. You do not author these. The verdict is one of:
  CONFIRMED         -> enters Discovery Ledger; build on it
  KNOCKOUT_SURVIVES -> gain not from the component you credited; re-hypothesize
  PLACEBO_KILL      -> likely noise / generic; abandon the mechanism story
  AXIS_MISS         -> real but unexplained; logged, not a confirmed mechanism
  SEED_FRAGILE      -> within seed noise; discard

## Read the Discovery Ledger every round
Each confirmed entry carries the axis it moves and a one-line design rule. Prefer
hypotheses that stack on confirmed mechanisms or attack an axis no confirmed entry
has moved yet. Do not re-litigate a mechanism the ledger already confirmed or killed.

## Agent-facing skills and evals
Use the project skills as narrow operating modes:
- autoalphafold3-researcher: choose one diagnostic target, one move family, and
  one falsifiable folding hypothesis.
- fold-cartographer: map scorer diagnostics to exactly one canonical target while
  keeping best_val_calpha_lddt as the only primary objective.
- autoalphafold3-trial-submit: validate one AutoFoldTrial JSON and return only
  the approved local orchestrator command.
- autoalphafold3-subagent-worker: generate bounded proposal artifacts for
  parallel hypothesis fanout; workers do not submit, integrate, call Modal, edit
  locked files, or write ledgers.

Skill evals must pass before autonomous research starts. Evals are synthetic and
local: no Modal calls, no GPUs, no hidden validation, no gate edits, and no real
trial submission. Run static/offline checks first, fan out sampled live evals on
small fast models, and reserve stronger adjudication for failed or ambiguous
outputs.

## You MAY edit
- autoalphafold3/ patch modules; selected nanofold/train model/training modules
- experiment configs, loss schedules, diffusion/noising/sampler wrappers
- optimizer/scheduler, recycling, curriculum, auxiliary heads, speed/memory options
- your own pre-registered Prediction

## You MAY NOT edit (hard locks)
- the validation split, scorer, raw labels, preprocessed features, result parser,
  baseline ledger, or autoalphafold3/modal_app.py
- the falsification controls: knock-out / placebo construction, gate thresholds
  (tau, rho, k_seed), verdict logic, or any Discovery Ledger entry
- you may not call modal run, spawn sandboxes, change GPU types, or raise caps
Touching any lock is evaluator hacking and invalidates the run.

## When stuck
- Three FAILs in one family -> change family, do not brute-force.
- If MANY candidates die at the gate for the SAME reason (e.g. repeated PLACEBO_KILL
  on attention-width changes), that pattern is itself a FINDING. Log it as a negative
  result in the ledger and move on; do not hide it.
- Two OOMs after a budget downgrade, or NaNs in both baseline and candidate paths ->
  request a single debug sandbox through the orchestrator.

## Framing (use consistently)
- "AlphaFold3-lite research sandbox", "NanoFold-style monomer folding".
- "falsification-gated discovery", "confirmed mechanism", "instructive kill".
- FORBIDDEN: "we beat / improve / reproduce / train AlphaFold3". We do not.
