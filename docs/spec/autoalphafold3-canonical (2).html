§ 0 Background — why this problem, why now

Protein folding is one of the cleanest scientific ML stories a hackathon can show on stage: an amino-acid sequence goes in, a three-dimensional backbone comes out, and the audience can see whether the predicted shape is plausible. It is high-impact because structure drives biology: proteins bind, catalyze, signal, misfold, and fail according to geometry.

But the target is not Google DeepMind's real AlphaFold3. Full AlphaFold3 is a large, production-grade biomolecular-complex model. It predicts joint structures for proteins, nucleic acids, small molecules, ions, and modified residues. Its official repository is an inference pipeline with parameter access and use constraints, not a hackathon-scale training substrate. Claiming that we improve, beat, reproduce, or train full AlphaFold3 would be false framing.

The right target is an AlphaFold3-lite research sandbox: small enough to read, train, profile, and patch; real enough that architectural and training changes mean something. That is why the implementation base is ogchen/nanofold. NanoFold is based on the AlphaFold2 and AlphaFold3 papers, is designed to train on a single mid-tier GPU, restricts the problem to monomer protein chains, predicts backbone atoms rather than full biomolecular complexes, and includes a real preprocessing stack for mmCIF structures, MSAs, templates, and Arrow IPC feature dumps.

Core hypothesis

GPT-5.5 can act as an autoresearcher for protein folding, discovering AlphaFold3-lite design improvements. auto-AlphaFold3 wraps a NanoFold-style AlphaFold3-lite implementation in an autoresearch loop: GPT-5.5 proposes architecture, training, loss, sampler, and optimization changes; Modal runs fixed-budget experiments through a hardened control plane; the locked evaluator returns validation C-alpha lDDT; the agent keeps improvements and logs the research trajectory.

We are using ogchen/nanofold because it is already an AlphaFold3-lite research sandbox, not because it matches real AlphaFold3. The project makes one honest claim: we run autoresearch over a small protein-folding model, under a locked benchmark, and search for design improvements.

0.1  The canonical twist: Fold Cartographer

The differentiated idea is not "Parameter Golf, but for proteins." Parameter Golf taught the compute pattern: smoke small, fan out exploration, drop to one sandbox for debugging, spike validation, and scale to zero when idle. auto-AlphaFold3 adds a folding-specific research layer: Fold Cartographer.

Fold Cartographer sits between the locked scorer and the GPT-5.5 agent. The primary score remains one number — best validation C-alpha lDDT — but the locked scorer also emits an aggregated geometry diagnostic report. Rather than slicing a tiny validation set into a noisy length × MSA-depth × region cross-product, it routes on four orthogonal, literature-grounded failure axes: local geometry, long-range topology, distogram-versus-coordinate decoupling, and stability/compute. This mirrors how the field already reasons about structure confidence — AlphaFold's own pLDDT (local geometry) versus PAE (confidence in the relative position of residue pairs, i.e. long-range topology), and CASP's emphasis on long-range contacts. The agent uses these signatures to choose the next move family. It does not optimize diagnostics directly; it optimizes locked validation C-alpha lDDT.

This is folding-specific and performance-relevant. A language-model compression run gets scalar loss. A folding run gets a 3D object. Every failure contains geometry: long-range contacts missing, local backbone unstable, pair representation learning contacts while the coordinate path fails, low-MSA examples underperforming, diffusion sampler too noisy, or recycling underused. Fold Cartographer turns those failures into routed research hypotheses.

0.2  Why Modal is still central

Modal is the elastic GPU lab, but the autoresearch agent does not design Modal infrastructure during the event. The compute architecture is predesigned from the Parameter Golf/autoresearch pattern: deploy the experiment app once, call many fixed-budget jobs through a typed runner, write per-trial artifacts to durable Volumes, use one debug sandbox when needed, and scale down while GPT-5.5 thinks. The folding agent spends its reasoning budget on protein-folding hypotheses, not cloud architecture.

§ 1 Autoresearch from first principles

Autoresearch is not "ask an LLM for ideas." It is a controlled loop:

  1. The agent reads the current repo, configs, logs, and constraints.
  2. It writes one hypothesis before editing anything.
  3. It makes one coherent model/training/sampler change.
  4. It submits a typed fixed-budget experiment to the orchestrator.
  5. The orchestrator runs preflight gates and spawns a Modal worker.
  6. The locked scorer returns validation C-alpha lDDT and diagnostics.
  7. Fold Cartographer turns diagnostics into a failure signature.
  8. The agent keeps, reverts, or diagnoses the change.
  9. The local orchestrator writes the canonical ledger entry.

For auto-AlphaFold3, the scalar is validation C-alpha lDDT. The editable surface is NanoFold-style architecture and training code. The fixed budget is a small training or inference-only sampler run on preprocessed Arrow features. The compute substrate is Modal. The locked contract is the validation split, scorer, raw labels, result parser, feature outputs, and baseline ledger.

Autoresearch contract

The agent may edit the model/training/sampler search surface, but not the benchmark contract or the Modal control plane. If it can change the scorer, split, validation labels, preprocessing outputs, result parser, or infrastructure guardrails, the project stops being research and becomes evaluator hacking. The exact editable and locked file lists are canonical in §4.

The research claim is the trajectory, not a single lucky checkpoint. A good run should read like this: "The agent saw that long-range contact precision improved while C-alpha lDDT stayed flat; it inferred the pair representation was learning distances but the coordinate path was not using them; it changed the geometry-loss ramp and recycling schedule; the score improved; it then tested whether the mechanism survived a seed rerun." That is an autoresearch story, not a grid search.

§ 2 Why auto-AlphaFold3 fits the shape

Folding-model autoresearch is heavier than probe autoresearch, but it has a stronger demo and a richer research surface. It fits the shape for six reasons.

1. AlphaFold3-lite gives meaningful architecture surface

NanoFold is not a black-box API. It exposes input embedding, MSA processing, template features, Pairformer-style reasoning, recycling, diffusion-style coordinate generation, distogram loss, diffusion/coordinate loss, training, dataset streaming, checkpointing, gradient checkpointing, and torch.compile. GPT-5.5 can make scientifically legible changes.

2. The metric is a scalar judges understand

Best validation C-alpha lDDT asks: how close is the predicted protein backbone geometry to the true structure, using alpha carbon positions? Higher is better. Loss curves, contact precision, runtime, memory, parameter count, NaNs, and OOMs are diagnostics. The scoreboard is one number.

3. Folding gives richer feedback than Parameter Golf

Parameter Golf has scalar compression loss. Folding gives geometry. The same C-alpha lDDT miss can be local loop instability, bad long-range packing, low-MSA failure, pair-contact failure, coordinate-head failure, or diffusion-sampler failure. Fold Cartographer gives the agent a domain-specific map of what went wrong.

4. Training trials are feasible with tiny configs and Modal

A full folding model is expensive. A NanoFold-style monomer model on a small fixed train subset is manageable. The event plan uses tiny crop sizes, reduced MSA depth, reduced trunk depth for smoke, short training windows, single-seed exploration, inference-only sampler trials, and final multi-seed validation only for the best candidate.

5. Diffusion sampler golf is a cheap inner loop

Because NanoFold has a diffusion module with inference-time schedule knobs, the agent can run inference-only sampler experiments on a fixed checkpoint. Sampler trials search diffusion steps, noise schedule, step scale, sample count, and label-free sample selection. They are far cheaper than retraining and directly tied to AlphaFold3-lite design.

6. Structure prediction creates visual artifacts

Protein folding produces shapes. The demo can show a baseline predicted backbone, the true backbone, the agent-improved prediction, the geometry error report, and the trajectory that got there. Even a modest numerical improvement is easier to sell when paired with a visible overlay and a coherent failure-to-fix mechanism.

SimplexFold as inspiration, not base

SimplexFold is useful because it shows the ambition bar: a bold folding architecture hypothesis, explicit higher-order geometric states, benchmark discipline, and trajectory-ready artifacts. auto-AlphaFold3 does not use SimplexFold as the implementation base. It uses NanoFold / AlphaFold3-lite as its base.

§ 3 The science, plainly

The biology can be explained without drowning judges in jargon. Use these definitions consistently.

Protein sequence input
A protein is a chain of amino acids. The sequence is the one-dimensional string. Folding predicts how that string sits in three-dimensional space.
Protein structure output
The 3D arrangement of the atoms in the protein. In this hackathon, we care mainly about monomer backbone geometry, not full complexes with ligands, DNA, RNA, or multiple protein chains.
C-alpha backbone simplified geometry
Each amino acid has an alpha carbon. Connecting those alpha carbon positions gives a clean trace of the protein backbone. It is less detailed than all-atom structure but excellent for fast scoring and visualization.
lDDT local distance difference test
A structure-quality score based on whether local distances in the predicted structure match local distances in the true structure. C-alpha lDDT applies this idea to alpha carbon positions. Plain English: how close the predicted protein backbone geometry is to the true structure, using the alpha carbon positions.
MSA multiple sequence alignment
A stack of related protein sequences from evolution. Correlated mutations can reveal which residues touch in 3D. Folding models use MSAs as evolutionary hints.
Templates known related structures
Existing protein structures that resemble the target sequence. Templates can help when a related fold has already been solved, but this hackathon benchmark disables them with max_templates=0 so every trial folds from the same no-template input contract.
AlphaFold2 vs. AlphaFold3 high level
AlphaFold2 popularized modern deep-learning protein structure prediction with MSA/template features, pair representations, Evoformer-style reasoning, recycling, and a structure module. AlphaFold3 broadened the target to biomolecular complexes and moved toward Pairformer-style reasoning plus diffusion-based coordinate generation. auto-AlphaFold3 borrows the AlphaFold3-lite flavor; it does not reproduce full AlphaFold3.
NanoFold / AlphaFold3-lite our sandbox
A small, hackable protein-folding implementation inspired by AlphaFold2 and AlphaFold3. It is monomer-focused, trainable on modest GPUs, and appropriate for agent-driven architecture/training/sampler search.
Fold Cartographer our twist
A locked diagnostic layer that converts folding failures into aggregated geometry signatures along four orthogonal axes: local geometry, long-range topology, distogram-versus-coordinate decoupling, and stability/compute. The local-versus-long-range split mirrors AlphaFold's pLDDT-versus-PAE confidence pair.
Design improvement what the agent discovers
A change to architecture, loss, training curriculum, optimizer, recycling, diffusion/noising, sampler, feature handling, or runtime that improves locked validation C-alpha lDDT without changing the evaluator or validation split.

§ 4 Methodology & search space

The agent should not be restricted to a single config file. That would make sense for a frozen-probe plan; it is too weak for folding architecture research. The agent may edit a broad but designated search surface: selected NanoFold model/training modules, an autoalphafold3/ patch area, configs, loss definitions, sampler wrappers, and training schedules. It may not edit scoring, split, preprocessing output, result parser, baseline ledger, or Modal control plane.

Search families

  • Architecture width/depth. Pair representation width, single representation width, number of trunk blocks, transition widths, head dimensions.
  • Pairformer / attention variants. Pair attention bias, triangle update simplifications, dropout placement, gated residuals, norm order, attention head count.
  • Diffusion / noising schedule. Training noise schedule, diffusion steps, loss weighting by noise level, train/sampling mismatch.
  • Diffusion sampler golf. Inference-only search over sampler steps, gamma_0, gamma_min, noise_scale, step_scale, s_max, s_min, p, number of stochastic samples, and label-free sample-selection rules.
  • Recycling. Number of recycles, random recycle range during training, recycle schedule, and early-run reductions for speed.
  • Auxiliary losses. Distogram/contact objectives, confidence heads, pLDDT-like diagnostics, alignment losses, and loss ramps.
  • Geometry losses. Coordinate loss, local-distance loss, bond/angle regularizers, backbone-only losses, clash proxies when cheap.
  • Distogram/contact losses. Distance-bin loss, contact precision diagnostics, harder contact weighting, long-range contact weighting.
  • Curriculum. Crop length, MSA depth, short-to-long sequence schedule.
  • Optimizer/scheduler. AdamW parameters, weight decay, warmup, cosine decay, gradient clipping, accumulation, batch size.
  • Memory/runtime optimization. bfloat16, gradient checkpointing, torch.compile toggles, batch sizing, model width reductions that permit more steps.
  • Feature handling. MSA dropout, feature normalization, empty-template placeholder behavior, MSA subsampling.

Fold Cartographer diagnostic targets

Before each new hypothesis, the agent reads the latest error_report.json and chooses exactly one of four orthogonal diagnostic targets. Each is one robust aggregate computable on a tiny validation set — not a fine-grained bucket cross-product — and each routes to a distinct move-family cluster:

  • local_geometry_weak (pLDDT-like; mean local lDDT, short-range contact precision) → try geometry loss, local backbone regularizers, termini/loop augmentation.
  • long_range_topology_weak (PAE-like; long-range contact precision, sequence separation ≥ 24) → try Pairformer/triangle updates, distogram weighting, crop curriculum.
  • distogram_good_lddt_flat (pair head learns contacts but the 3D path does not follow) → try coordinate loss, recycling, diffusion sampler.
  • stability_compute (NaN/OOM rate, runtime, peak memory) → try width/depth, gradient clipping, checkpointing, torch.compile.

Sequence length and MSA depth are reported as two or three coarse buckets for context only; the agent does not route on them during search, because per-bucket estimates on a small validation set are too noisy to steer hypotheses.

Decision rule

During search, a candidate is a KEEP if it improves best validation C-alpha lDDT over the current best by a minimum threshold and does not create unacceptable NaN/OOM/runtime behavior. A candidate is DISCARD if it fails the metric, violates budget, or only improves by noise. A candidate is FAIL if the training path breaks. A candidate is INFRA_FAIL if the Modal worker or artifact path fails independent of the patch.

During broad search, single-seed noise is tolerated. During final validation, the best candidate gets multiple seeds on the fixed public validation split, plus optional hidden validation if staged before the event. The hidden split is final-only; it is not used for search diagnostics.

Corrected Modal contract

The Modal compute architecture is creator-designed. The agent submits AutoFoldTrial JSON objects through the local orchestrator. It may not call modal run directly, spawn arbitrary sandboxes, modify Volumes, change GPU types, raise max_containers, or edit autoalphafold3/modal_app.py during search.

5.1  Control-plane / worker split

The platform has three layers:

  1. GPT-5.5 agent. Reads repo, configs, logs, and diagnostics. Writes one hypothesis, one patch, one AutoFoldTrial.
  2. Local orchestrator. Validates patch scope, checks forbidden files, enforces budget/resource tier, runs preflight, submits to deployed Modal function, polls results, writes canonical ledger.
  3. Modal workers. Trial/sampler workers load preprocessed Arrow features, apply the patch in an isolated workspace, and write predictions/checkpoints under trial-scoped /runs paths. A separate scorer-only worker mounts locked labels/manifests, computes metrics, and returns canonical diagnostics.

This separation keeps the scientific search broad while narrowing the operational surface. The agent can make real architecture/training/sampler changes, but it cannot mutate the infrastructure that enforces cost, locking, and scoring.

5.2  Deploy once, call many

The Modal app is deployed once before the search loop. Trials are launched through modal.Function.from_name(...).spawn(...) or .remote(...). Do not issue many concurrent modal run processes. This avoids app teardown conflicts, keeps images and Volume mounts consistent, and makes job polling deterministic.

autoalphafold3/orchestrator.py — structural sketch

import json
import time
from pathlib import Path
import modal

APP_NAME = "autoalphafold3"
LEDGER = Path("runs/ledger.jsonl")
FORBIDDEN = {
    "autoalphafold3/modal_app.py",
    "autoalphafold3/locked_scorer.py",
    "autoalphafold3/result_parser.py",
    "locked/",
    "runs/baseline/",
}

def preflight(trial: dict) -> None:
    check_forbidden_files_unchanged(FORBIDDEN)
    check_config_schema(trial["config_path"])
    check_patch_scope(trial["patch_path"])
    check_param_count_under_cap(trial)
    run_tiny_forward_pass(trial)
    run_one_batch_loss_check(trial)
    run_scorer_dry_run_schema_check()

def submit_trial(trial: dict) -> str:
    preflight(trial)
    fn = modal.Function.from_name(APP_NAME, "run_trial")
    call = fn.spawn(trial)
    return call.object_id

def poll_call(call_id: str, timeout_s: int = 3600) -> dict:
    call = modal.FunctionCall.from_id(call_id)
    start = time.time()
    while time.time() - start < timeout_s:
        try:
            return call.get(timeout=0)
        except TimeoutError:
            time.sleep(10)
    return {"status": "INFRA_FAIL", "reason": "timeout", "call_id": call_id}

def poll_scored_trial(run_call_id: str) -> dict:
    artifact = poll_call(run_call_id)
    if artifact.get("status") == "INFRA_FAIL":
        return artifact
    score_fn = modal.Function.from_name(APP_NAME, "score_trial")
    score_call = score_fn.spawn(artifact["trial_id"])
    metrics = poll_call(score_call.object_id)
    append_ledger({"trial_id": artifact["trial_id"], "artifact": artifact, "metrics": metrics})
    return metrics

def append_ledger(row: dict) -> None:
    LEDGER.parent.mkdir(parents=True, exist_ok=True)
    with LEDGER.open("a") as f:
        f.write(json.dumps(row, sort_keys=True) + "\n")

5.3  Agent-accessible tools

  • GPT-5.5 / Codex / goal mode. Researcher and code editor. It proposes hypotheses and writes patches.
  • Local orchestrator. The only approved way to submit experiments.
  • Git. Every KEEP is a commit. The commit message starts with move family and diagnostic target.
  • Fold Cartographer reports. Aggregated diagnostics used to route the next hypothesis.
  • Locked scoring. The agent receives metrics and diagnostic summaries. It does not edit the scorer.
  • Cost caps and failure statuses. NaNs, OOMs, shape errors, timeouts, and infra failures are first-class outcomes.

5.4  Storage discipline

Workers never append to one shared ledger. Each trial worker writes into its own directory: /runs/trials/T123/artifact_manifest.json, training_log.json, stdout.log, stderr.log, patch.diff, checkpoint, and optional overlay-ready coordinates. The scorer-only worker reads that artifact directory and returns metrics plus an error_report payload; the local orchestrator merges those returned diagnostics into runs/ledger.jsonl. This avoids Volume write contention and keeps postmortems clean.

5.5  Preflight gates

  • Patch-scope check: forbidden files unchanged.
  • Config schema check: all required NanoFold keys present.
  • Parameter-count check: under cap.
  • Tiny forward pass: no shape error.
  • One-batch loss check: finite total_loss, diffusion loss, and distogram loss.
  • Scorer dry-run: canonical JSON schema valid.
  • Only then launch fixed-budget training or sampler evaluation.

§ 6 Data and benchmark

Data is the make-or-break constraint. Full folding preprocessing is too slow and too failure-prone to discover on event day.

Data rule

Preprocess before event. During the event, never run full MSA/database preprocessing except tiny smoke checks. The hackathon starts from preprocessed Arrow IPC features, a tiny train subset, a fixed public validation split, and a working NanoFold default/tiny baseline.

6.1  NanoFold data path

NanoFold consumes PDB/mmCIF structural inputs, searches sequence databases for MSAs, can search template databases, and writes Arrow IPC feature dumps. The relevant preprocessing for this hackathon includes mmCIF parsing from PDB, small BFD / Uniclust30-style sequence search for MSAs, empty template placeholders, and Arrow IPC feature output. This is real protein-folding data work, not something to debug after kickoff.

No-template benchmark

Pin max_templates=0 for all official runs. A template is a previously solved structure of a similar protein. Allowing templates changes the task: the model can lean on an external solved-shape hint instead of folding from sequence, MSA/evolutionary signal, and learned geometry alone. It also makes setup heavier because a template database must be provisioned, searched, versioned, and locked.

The simpler and cleaner hackathon contract is no-template NanoFold: template tensors may exist as empty schema placeholders, but the event does not require template DB access, template search, template-sensitive diagnostics, or agent control over template retrieval. This reduces leak risk and keeps every baseline and agent trial on the same input contract.

6.3  Locked benchmark wrapper

NanoFold's default config has a train split ratio. 6.4  Splits

  • Train subset. Fixed small subset of preprocessed monomer chains for fast experiments. Size is chosen by runtime, not ambition.
  • Public validation subset. Fixed, small, fully locked. The agent sees metrics and aggregated diagnostics, not labels or split construction.
  • Baseline ledger. Contains NanoFold default/tiny baseline metrics, immutable after T+1.

6.5  Metric and diagnostics

The primary scalar is best validation C-alpha lDDT. Diagnostics are the four orthogonal Fold Cartographer axes — local geometry (local lDDT, short-range contact precision), long-range topology (long-range contact precision), distogram-versus-coordinate gap, and stability/compute (runtime, memory, parameter count, NaN/OOM rate) — plus coordinate/distogram loss and final seed variance. lDDT by coarse length bucket is reported for context only, never routed on. Diagnostics are emitted by the locked scorer and summarized by Fold Cartographer.

6.6  Default/tiny baseline

The first baseline is the upstream NanoFold default path on the smallest feasible setup. The second is a local auto_tiny config that preserves the training/evaluation path while shrinking crop length, MSA depth, trunk depth, diffusion steps, batch size, and train steps for fast smoke experiments. The public scoreboard compares agent candidates against the best baseline available by T+1.

§ 7 Where Modal comes in

Modal is the elastic GPU lab for autonomous folding experiments. The deeper claim is sharper than "we used Modal for GPUs." Parameter Golf proved Modal can elastically scale how much compute an agent uses; auto-AlphaFold3 proves it can elastically scale what kind. A single folding-research session naturally spans five sharply different execution modes, and the agent moves between them with small orchestration changes. NanoFold-lite is primarily independent single-GPU training and inference, so this plan uses parallel independent single-GPU jobs by default; multi-GPU is useful only if the training code actually runs distributed execution, which it does not here.

The five-mode + routing thesis

Folding autoresearch demands five distinct execution modes in one session: a massive parallel CPU fan-out for data, cheap single-GPU training trials, a few-hundred-job inference-only sampler burst, a one-Sandbox debug drop, and a final validation/render spike — then scale to zero while GPT-5.5 thinks. Each maps to specific Modal primitives and a budget tier from schema.py. The original autoresearch run hit roughly 113 experiments across 238 GPU-hours on a near-uniform language-model task; folding's work is not uniform, which makes it a stronger demonstration of Modal's actual value.

7.1  Execution modes

MODE A
Parallel CPU fan-out — build the benchmark
Function.map · Volume.commit · pre-event · CPU · scale → 0

mmCIF parsing and MSA search over hundreds of monomer chains is embarrassingly parallel, heavy, and one-shot. Template fields are written as empty placeholders because the benchmark is pinned to max_templates=0. One preprocess_chain.map(all_ids) wave builds the Arrow feature set into the data Volume, then releases. Parameter Golf had no such phase — its data was plain text.

This is the feasibility hinge: it runs before T+0. During the event, trials read cached features; full database preprocessing is banned except tiny smoke checks. Tier: pre-event prep (§8).

MODE B
Single-GPU training trials — broad search
spawn → FunctionCall.get · gpu="A100-80GB" (pinned) · max_containers 2–6

The orchestrator submits typed AutoFoldTrial jobs through Function.from_name(...).spawn(...) and polls FunctionCall.get(timeout=0) via the status Dict — the documented job-queue pattern, results retained for up to seven days. Independent single-GPU trials fan out in parallel. All official trials run on one pinned A100-80GB Function: 80 GB covers the tiny no-template crops, and a single fixed tier keeps the runtime and peak-memory numbers in the stability_compute diagnostic comparable across trials. A mixed-tier fallback list would let the scheduler vary the GPU and confound those comparisons, so it is avoided for scored trials.

Tier: trial. The main loop.

MODE C
Inference-only sampler burst — the signature
Function.map · gpu="A100"/"L40S" · max_containers ~50 · scaledown_window 20s

Because the diffusion sampler runs on a frozen checkpoint, hundreds of sampling configs — step counts, noise schedules, seeds — fire in one sample_once.map(grid) wave, each a few seconds long. Modal spikes the container pool toward fifty and collapses to zero in about a minute. Mounts are read-only and jobs return label-free sample artifacts/signals rather than each committing to the Volume, which avoids concurrent-commit contention and keeps validation labels out of the sampler path. The wave of samples is the conformational ensemble the demo renders.

Tier: sampler. The cheapest, most parallel, most visible Modal moment on stage; it has no analog in Parameter Golf, which has no cheap parallel inference loop.

MODE D
Interactive debug Sandbox — triggered drop
Sandbox.create(image, gpu) · one at a time · exec inspect

Open one debug Sandbox only on explicit triggers: three consecutive shape errors in the same family, two OOMs after a budget downgrade, NaNs in both baseline and candidate paths, or scorer failure on a previously valid checkpoint. modal.Sandbox.create() provisions an exec-able GPU container with the training image in seconds; inspect the checkpoint, shape, or OOM, then terminate.

Tier: debug. Drops are orchestrator/human-triggered, not agent-discretionary, and serial by design — no point parallelizing a sequential bug hunt.

MODE E
Validation spike — multi-seed + render, then zero
fan-out candidates × seeds · gpu="H100" · render overlays · scaledown_window 60s

Final-only. At T+7, stop broad search; rerun the best one to three candidates across multiple seeds on the locked public split; use hidden validation only here; render predicted-vs-true backbone overlays for baseline and best candidate. Scale up briefly, validate honestly, scale down.

Tier: final. The only place hidden validation is touched.

After each trial the scorer-only function returns canonical metrics plus an error_report payload; Fold Cartographer summarizes the failure signature, and the agent must choose one diagnostic target before proposing the next patch. Metrics and diagnostics return to the orchestrator (Fig. 7), so the loop reasons over geometry failure modes, not a single scalar, and the scorer never needs writable access to the trial artifact mount.

7.4  How the agent's code edits reach Modal

A deployed Modal function runs the code captured when it was deployed, so the loop needs a disciplined answer to one question: how does a GPT-5.5 patch to NanoFold reach the GPU container — fast, and reproducibly? Separate the slow-changing image from the fast-changing code, then run the loop at two rates: redeploy only on a code change, and spawn/map freely on config-only changes.

The rule

Pin the image (CUDA base + exact library versions) so heavy layers cache and stay reproducible. Add the editable dirs with copy=False so they mount at container startup instead of baking an image layer — a code patch then re-uploads only the mount, not a layer. Redeploy when the editable-surface git SHA changes; otherwise just submit more jobs against the current deployment.

Two rates, one chokepoint

CODE
Editable surface changed

Preflight → modal deploy (copy=False mount re-upload, seconds) → record git SHA + app revision. Because spawn targets a deployed function, deploy is the single point every code change passes through — which is exactly where the hardened preflight sits. Forbidden-files, scorer dry-run, and schema validation run before deploy, so a patch touching the locked surface can never ship to a worker.

CONFIG
Config / seed / schedule only

No deploy. Route to the current deployed trial Function and call spawn(...), or call sample_once.map(...) against the current deployment. Most of a sweep changes only config, so the expensive step fires rarely while cheap fan-outs run freely.

DEBUG
Mode D exception

No deploy. A Sandbox with the current committed SHA and live copy=False mount gives fast inspection cycles while hunting a NaN. It is human-triggered, read-only against /runs, and the agent may not apply new patches inside the Sandbox outside the orchestrator wave.

autoalphafold3 orchestrator — deploy on code, spawn on config, freeze per wave

EDITABLE = ["nanofold", "autoalphafold3/runner.py", "autoalphafold3/sampler.py"]
last_deployed_sha = None
last_deployed_app_rev = None

def run_wave(proposal):
    global last_deployed_sha, last_deployed_app_rev
    apply_patch(repo, proposal.patch)              # mutate working tree ONCE, before launch
    preflight(repo, proposal)                      # forbidden-files + scorer dry-run + schema
                                                   #   -> abort the wave on any failure

    sha = git_commit(repo, EDITABLE, proposal)     # content-addressed code snapshot
    if sha != last_deployed_sha:                   # did the editable surface change?
        last_deployed_app_rev = sh("modal deploy autoalphafold3/modal_app.py")   # copy=False -> mount re-upload
        last_deployed_sha = sha
    # else: pure config/seed sweep -> no deploy

    fn = modal.Function.from_name("autoalphafold3", "run_trial")   # one pinned trial Function
    calls = [fn.spawn(t) for t in proposal.trials]                 # fan out the frozen wave
    artifacts = [poll(c) for c in calls]                           # each trial wrote /runs/trials/{id}
    score_trial = modal.Function.from_name("autoalphafold3", "score_trial")
    score_calls = [score_trial.spawn(a["trial_id"]) for a in artifacts]
    metrics = [poll(c) for c in score_calls]                       # scorer-only function sees labels
    record(proposal, sha=sha, app_rev=last_deployed_app_rev,
           scorer_version=metrics[0]["scorer_version"],
           features_hash=metrics[0]["features_hash"])
    return keep_or_revert(metrics)                 # only now may the next patch be applied

Why freeze the tree per wave

With copy=False, a container reads the editable dirs at startup. If the orchestrator patched source while a wave's containers were still launching, jobs in the same wave could run different code and the keep/revert decision would be meaningless. So the working tree is mutated exactly once per wave — apply patch, commit, deploy-if-changed, launch, wait — and the next patch waits until the wave resolves.

The strongest Modal story is not peak GPU count. It is elasticity across both amount and kind: a one-shot data fan-out, durable Volumes with background commits, independent trials, an inference-only sampler burst, one-Sandbox debugging, a bounded validation spike, and scale-to-zero while the agent reasons over geometry diagnostics. Every trial records the exact code, app revision, scorer, and features it ran against, so the research trajectory is reproducible, not just fast.

    Use auto_tiny, short budgets, sampler-only inner loops, and final multi-seed only for best candidates.
  • Agent mutates infrastructure. If the agent edits Modal app, scoring, or budget caps, the loop becomes unsafe.
    Modal app is locked. Agent submits typed trials only. Orchestrator checks forbidden files before every run.
  • Default NanoFold random split violates benchmark lock. A ratio split is not a locked evaluator.
    Use explicit manifests and manifest hashes through locked_dataset.py. Refuse official trials using random split.
  • Loss-only evaluation lacks prediction C-alpha lDDT. NanoFold's training path reports losses; the demo needs structures and C-alpha lDDT.
    Implement separate prediction-artifact generation and scorer-only lDDT: checkpoint load, inference/sample path, C-alpha extraction, locked scorer call, and overlay-ready coordinates stay separated by the Volume boundary.
  • Agent creates unstable architecture. Broad editable surface can break tensor shapes.
    Patch through scaffolded patterns where possible. Add shape tests, param-count checks, tiny forward pass, and automatic rollback for failure states.
  • NaNs and OOMs dominate the ledger. Diffusion losses and mixed precision can destabilize.
    Treat NaN/OOM as FAIL. Log honestly, downgrade budget only within rules, and use one debug sandbox under trigger rules.
  • Validation C-alpha lDDT is noisy. Small validation subsets can produce false positives.
    Use a keep threshold above baseline noise. Rerun final candidates across multiple seeds and report mean/variance.
  • Diagnostics leak labels or become objective hacking. Rich diagnostics can tempt overfitting to public validation.
    Diagnostics are aggregated, locked, and public-val only. Hidden validation remains final-only. Primary metric stays best validation C-alpha lDDT.
  • Shared ledger writes collide. Multiple workers appending one file can lose data.
    Workers write only per-trial directories. Local orchestrator is the single ledger writer.
  • Modal GPU image issues. Torch, CUDA, pyarrow, and NanoFold dependencies can fail inside the container.
    Build and test image before event. Pin dependencies. Keep a minimal fallback image that runs tiny baseline and scorer.
  • Unfair "AlphaFold3" overclaim. Judges may infer we modified real AlphaFold3.
    Repeat the framing everywhere: NanoFold-style AlphaFold3-lite sandbox, not full AlphaFold3.
  • SimplexFold comparison distracts from NanoFold base. The project could drift into explaining another repo.
    Use SimplexFold only as ambition/demo reference. All implementation, metrics, and patches live in NanoFold / auto-AlphaFold3.
  • Demo does not improve score. The final candidate may only match baseline or improve modestly.
    Build the demo around trajectory, diagnostics, hypotheses, failure signatures, Modal jobs, and overlays. A modest improvement with mechanism still lands.

# auto-AlphaFold3 — GPT-5.5 autoresearch for AlphaFold3-lite folding

## Core hypothesis

GPT-5.5 can act as an autoresearcher for protein folding, discovering
AlphaFold3-lite design improvements.

We are NOT modifying, reproducing, training, improving, or beating Google
DeepMind's real AlphaFold3. We are using ogchen/nanofold as a small
NanoFold-style AlphaFold3-lite research sandbox.

## Objective

Maximize best validation C-alpha lDDT under the locked benchmark.

C-alpha lDDT means: how close the predicted protein backbone geometry is to
the true structure, using alpha carbon positions.

Primary metric:
    best_val_calpha_lddt

Diagnostics (four orthogonal Fold Cartographer axes + context):
    local_geometry:    local_lddt, contact_precision_short
    long_range:        contact_precision_long      # CASP separation >= 24
    distogram_vs_3d:   distogram_coordinate_gap
    stability_compute: runtime_seconds, peak_gpu_memory_gb, param_count, nan_oom_status
    losses:            val_loss, diffusion_loss, distogram_loss, coordinate_loss
    context_only:      calpha_lddt_by_length       # coarse buckets; not routed on
    final_only:        seed_variance_for_final_validation

## Diagnostic routing rule

Before each new hypothesis, read the latest `error_report.json`.

Choose exactly one of four orthogonal diagnostic targets:
- local_geometry_weak        # pLDDT-like: local lDDT, short-range contacts
- long_range_topology_weak   # PAE-like: long-range contact precision (separation >= 24)
- distogram_good_lddt_flat   # pair head learns contacts but 3D does not follow
- stability_compute          # NaN/OOM rate, runtime, peak memory

Your next intervention must explain how it addresses that target.

Do not optimize diagnostics directly. The only primary metric is
best validation C-alpha lDDT.

## The loop

For each trial:

1. Read current best result, latest error_report.json, and last 20 ledger entries.
2. Choose exactly ONE diagnostic target.
3. Choose exactly ONE move family.
4. Write a hypothesis BEFORE editing code.
5. Predict the expected effect on validation C-alpha lDDT and diagnostics.
6. Patch the allowed model/training/sampler surface.
7. Create one AutoFoldTrial JSON.
8. Submit through the approved orchestrator command.
9. Read canonical metrics and Fold Cartographer report.
10. Decide KEEP, DISCARD, FAIL, or INFRA_FAIL.
11. Commit KEEP runs.
12. Revert DISCARD runs.
13. Log FAIL and INFRA_FAIL honestly.
14. Write a postmortem.

## Decision rule

KEEP if:
- best_val_calpha_lddt improves over current best by the configured threshold
- no NaN/OOM/shape failure
- runtime and memory are within cap
- parameter count is within cap
- improvement is not explained by invalid evaluation

DISCARD if:
- score does not improve enough
- diagnostics regress badly without score gain
- patch is too expensive
- patch is incoherent or too broad

FAIL if:
- NaN
- OOM
- shape error
- timeout
- scorer cannot run
- checkpoint cannot be loaded

INFRA_FAIL if:
- Modal function fails before running trial logic
- Volume unavailable
- image/dependency issue
- orchestrator cannot poll result

A FAIL is not shameful. A hidden or misreported FAIL is unacceptable.

## Git discipline

For DISCARD:
    git checkout -- <changed files>

For FAIL:

    revert code unless the fix is a minimal model/training repair in allowed files

## Stuck rule

If 8 consecutive trials fail to improve best_val_calpha_lddt:

1. Stop making small variations of the same move.
2. Switch to a move family not used in the last 20 trials.
3. Choose a different diagnostic target if the current target has been tried 4 times.
4. Write why the switch is rational.

Examples:
- Contact precision improves but lDDT does not: try geometry_loss, recycling, or sampler.
- Loss falls but lDDT is flat: try auxiliary_loss balance or curriculum.
- OOMs dominate: try memory_runtime or width_depth reduction.
- Low-MSA examples underperform: try feature_handling or curriculum.
