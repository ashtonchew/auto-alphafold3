§ 0Demo Goal

This is a running planning document for the eventual auto-AlphaFold3 demo frontend. It is not a visual design spec. It defines what the demo should prove, what the UI should contain, what metrics need to be available, and what backend work must exist before each screen can be honest.

One story

We did not improve Google DeepMind AlphaFold3. We gave GPT-5.4 mini a locked NanoFold-style AlphaFold3-lite research sandbox, a fixed C-alpha lDDT benchmark, Fold Cartographer diagnostics, a falsification gate, and an elastic Modal lab. GPT-5.5 remains an explicit escalation model. In the one-hour practical run, the system proposes frozen-checkpoint sampler mechanisms, tests them, kills weak explanations, and retains only confirmed sampler improvements.

The demo should not feel like a leaderboard screenshot or a static notebook. It should feel like watching an autoresearch loop make and test scientific claims.

The audience should leave with five concrete takeaways

  1. The benchmark is locked and honest.
  2. The agent is making pre-registered, falsifiable hypotheses.
  3. Fold Cartographer turns folding failures into actionable geometry axes.
  4. Modal provides the elastic execution modes needed to test those hypotheses.
  5. The final result is a research trajectory: confirmed mechanisms, killed candidates, metrics, artifacts, and overlays.

§ 1Primary Narrative

The main story arc should be linear enough for a judge to follow on stage, but detailed enough that the repository and artifacts still support agent review later.

  1. Baseline. Start with the NanoFold default/tiny baseline on the fixed no-template public validation split.
  2. Diagnosis. Show the current Fold Cartographer failure signature, such as weak long-range topology or a distogram/coordinate gap.
  3. Hypothesis. Show the agent's pre-registered mechanism: what it changed, which diagnostic axis should move, and what score effect it predicts.
  4. Experiment. Show the 20-candidate Modal sampler burst running under fixed budget and locked mounts.
  5. Verdict. Show the result, the falsification gate verdict, and whether the candidate was confirmed, discarded, killed by placebo/seed fragility, failed, or infra-failed.
  6. Accumulation. Show the Discovery Ledger as the durable set of confirmed mechanisms and instructive kills.
  7. Final comparison. Show baseline vs. best confirmed candidate on metrics and structure overlay.
Strong version

The strongest one-hour version of the demo is not "trial 12 got a higher score." It is "trial 12 claimed fewer diffusion steps with a lower step scale would reduce sampler noise and close the distogram/coordinate gap; the gap closed, lDDT improved, removing the sampler change removed the gain, a matched sampler placebo did not reproduce it, and the result survived seed checks."

§ 2Stage Wireframe

The first screen should read as an evidence board, not a marketing page. The hero content is the live run state: current score, trajectory, mechanism verdict, geometry diagnostics, Modal activity, and a structure overlay.

§ 3UI Surfaces

01 · Overview

Run Overview

Purpose. Orient the audience in one screen.

Content
  • Current best best_val_calpha_lddt
  • Baseline best_val_calpha_lddt
  • Delta vs. baseline
  • Number of submitted trials
  • Counts by status: KEEP, DISCARD, FAIL, INFRA_FAIL, falsification-gate verdicts
  • Current best candidate ID and git SHA
  • Active split, scorer version, feature hash, manifest hashes
  • Modal asset status: data Volume ready, locked Volume ready, scorer-only boundary intact
Prerequisites
  • Frozen baseline metrics under runs/baseline/
  • Canonical runs/ledger.jsonl
  • Metrics payloads with scorer and manifest hashes
  • Modal asset audit output persisted or queryable
02 · Trajectory

Trial Trajectory

Purpose. Show autoresearch as a sequence, not a single result.

Content
  • X-axis: trial number or time; in the one-hour plan this is the 20-candidate sampler burst plus one gated candidate
  • Y-axis: best_val_calpha_lddt
  • Baseline line
  • Point color: move family
  • Point shape or outline: trial status and falsification verdict
  • Hover/details panel: hypothesis, diagnostic target, prediction, intervention, lDDT, delta, runtime, memory, postmortem
  • Visible failed and killed candidates, not hidden
Prerequisites
  • Append-only ledger with every trial and gate-control result
  • Standardized status/verdict fields
  • Trial postmortems
  • Move family and diagnostic target recorded for every trial

Important statuses

  • CONFIRMED or KEEP: score improved and mechanism survived the gate
  • DISCARD: valid run but no meaningful improvement
  • FAIL: code/model/training/scorer artifact failure
  • INFRA_FAIL: Modal, Volume, polling, image, dependency, quota, or storage failure
  • SEED_FRAGILE: improvement did not survive seed noise
  • PLACEBO_KILL: unrelated placebo reproduced the gain
  • KNOCKOUT_KILL: removing the credited mechanism did not remove the gain
Realized mockup

Interactive trajectory with a selected-trial detail panel: ui/panels/02-trial-trajectory.html.

03 · Hypothesis

Hypothesis Card

Purpose. Prove the agent made a falsifiable claim before seeing the result.

Content
  • Trial ID and candidate ID
  • Diagnostic target: one of local_geometry_weak, long_range_topology_weak, distogram_good_lddt_flat, stability_compute
  • Move family
  • Hypothesis, intervention, and prediction
  • Expected metric movement: primary best_val_calpha_lddt, targeted diagnostic axis, expected runtime/memory risk
  • Patch summary and linked diff
  • Config summary and budget tier
Prerequisites
  • Trial JSONs in trials/T###.json
  • Patch diffs in trial artifact directories
  • Ledger fields for pre-run hypothesis and prediction
  • Stable candidate IDs
04 · Cartographer

Fold Cartographer Panel

Purpose. Show why this is folding-specific research rather than scalar hill-climbing.

Content
  • Four diagnostic axes: local geometry, long-range topology, distogram vs. coordinate realization, and stability/compute
  • Latest failure signature
  • Axis movement for selected trial vs. baseline
  • Context-only buckets: coarse length and MSA-depth summaries, clearly marked as not routing targets
  • Explanation of the chosen next target
Prerequisites
  • Real Fold Cartographer report, not toy signatures
  • Scorer output includes contact precision and local/long-range geometry diagnostics
  • Runner captures runtime, memory, parameter count, NaN/OOM/timeout
  • Baseline diagnostics exist for comparison
05 · Gate

Falsification Gate Panel

Purpose. Demonstrate that mechanism claims are tested, not merely asserted.

Content
  • Candidate claim
  • Gate controls: knockout/removal of credited change, placebo or unrelated-control comparison, seed rerun or seed-noise check
  • Verdict numbers: candidate delta vs. baseline, knockout delta, placebo delta, seed mean and variance, named diagnostic-axis movement
  • Final verdict: confirmed, killed, fragile, inconclusive
Prerequisites
  • Gate-control job definitions
  • Orchestrator-owned gate logic
  • Result schema for control runs
  • Discovery Ledger entries for confirmed mechanisms and killed candidates
Realized mockup

Falsification gate with control bars and verdict numbers: ui/panels/05-falsification-gate.html.

06 · Discovery

Discovery Ledger

Purpose. Show reusable research findings, not just checkpoints.

Content
  • Confirmed mechanisms with one-line design rule
  • Killed candidates with reason
  • For each entry: trial ID, git SHA, scorer version, feature/manifest hashes, diagnostic axis, primary metric delta, gate verdict, linked artifacts
Prerequisites
  • Separate Discovery Ledger or ledger subtype
  • Gate verdicts
  • Provenance fields in every scored result
  • Human-readable mechanism summaries
07 · Modal

Modal Execution Panel

Purpose. Make the Modal story legible without making the demo about infrastructure internals.

Content
  • Current or latest execution mode: smoke, trial, sampler burst, debug sandbox, final validation, gate control wave
  • Job count and status
  • GPU type and fixed resource tier
  • Container count over time if available
  • Volume mounts shown as policy, not secrets: public features read-only, trial runs read/write under trial directory, locked labels scorer-only
  • Cost/time summary
Prerequisites
  • Modal deployed app
  • Function call IDs stored in ledger/artifacts
  • Job polling/status capture
  • Resource tier metadata in result payloads
  • Optional Modal dashboard deep links
08 · Overlay

Structure Overlay

Purpose. Connect the scalar and diagnostic story to visible protein geometry.

Content
  • True C-alpha backbone
  • Baseline prediction
  • Best confirmed candidate prediction
  • Optional per-residue or segment error coloration
  • Selected target metadata: target ID, length, MSA depth bucket
  • Before/after lDDT and targeted-axis movement
Prerequisites
  • Prediction artifacts with C-alpha coordinates
  • Baseline prediction artifacts
  • Scorer-only labels available to rendering path only when allowed
  • Overlay renderer can consume real artifacts, not sample traces
  • Target selection rule that does not cherry-pick dishonestly
09 · Sampler

Sampler Burst / Ensemble Moment

Purpose. Show the live Modal moment that is unique to folding.

Content
  • Frozen checkpoint
  • Grid of sampler settings
  • Number of samples
  • Runtime and container burst
  • Label-free sample selection rule
  • Returned ensemble visualization or summary
  • Locked scorer result after selection
Prerequisites
  • Inference-only sampler harness
  • Label-free selection rules
  • sample_once.map(...) or equivalent Modal burst
  • Ensemble artifact format
  • Scoring path that happens only after samples are produced

§ 5Demo Thresholds

Minimum honest demo

The minimum frontend worth showing needs:

  1. Baseline metrics and diagnostics.
  2. At least three valid trial rows in the trajectory.
  3. At least one failed or discarded trial shown honestly.
  4. At least one candidate with a pre-registered hypothesis and postmortem.
  5. Fold Cartographer report with real, non-toy diagnostic axes.
  6. A baseline-vs-candidate metric table.
  7. A structure overlay from real prediction artifacts, or clearly labeled sample overlay if real predictions are not ready.
  8. Modal job evidence for at least one execution mode.
Boundary

Without real prediction artifacts and scorer-only C-alpha lDDT, the frontend can only be a scaffold/demo shell.

Strong demo

A strong demo adds:

  1. One confirmed mechanism that survives the Falsification Gate.
  2. One killed candidate that looked good by scalar score but failed a control.
  3. A sampler burst with visible Modal fanout.
  4. Multi-seed final validation for the best confirmed candidate.
  5. Before/after overlay where the named diagnostic axis visibly matches the metric story.
  6. Discovery Ledger panel with confirmed mechanisms and instructive kills.

§ 6Backend Readiness Checklist

Before building beyond static mockups:

  • Real runs/baseline/metrics.json exists.
  • Real runs/baseline/error_report.json exists.
  • runs/ledger.jsonl receives orchestrator-written trial rows.
  • Trial JSONs exist under trials/.
  • Modal app is deployed and callable by Function.from_name(...).spawn(...).
  • run_trial writes real trial artifacts from cached Arrow features.
  • score_trial returns scorer-only public validation metrics.
  • Prediction artifacts include C-alpha coordinates.
  • Fold Cartographer emits the four canonical axes.
  • Runtime, memory, parameter count, NaN/OOM/timeout are captured.
  • Falsification Gate jobs and verdict schema exist.
  • Discovery Ledger exists or is derivable from canonical ledger rows.
  • Overlay renderer consumes real artifacts.
  • Sampler burst harness exists for frozen-checkpoint inference.
  • Final validation path can run multi-seed candidates.

§ 7Frontend Data Contracts To Design Next

The frontend should eventually read stable JSON contracts rather than scraping arbitrary logs.

Candidate inputs

  • runs/ledger.jsonl: append-only trial and verdict rows
  • runs/baseline/metrics.json: immutable baseline metrics
  • runs/baseline/error_report.json: immutable baseline diagnostics
  • runs/discovery_ledger.jsonl: confirmed mechanisms and killed candidates
  • runs/trials/<trial_id>/artifact_manifest.json: artifact pointers
  • runs/trials/<trial_id>/metrics.json: scored metrics
  • runs/trials/<trial_id>/error_report.json: Fold Cartographer report
  • runs/trials/<trial_id>/falsification/*.json: control-run verdicts
  • runs/trials/<trial_id>/overlay/*.json: coordinates or render metadata

Open schema questions

  • Should gate-control runs be normal ledger rows with a control_for field, or nested under the candidate trial?
  • Should Discovery Ledger be a separate file or a filtered view over canonical ledger rows?
  • Should frontend-ready summaries be generated as a build step, such as runs/ui_state.json, to avoid reading many trial directories?
  • How should hidden/final validation be displayed without exposing labels or implying it was used during search?
  • What target-selection rule should choose overlay examples without cherry-picking?

§ 9Non-Goals

  • Do not build a landing page as the main demo.
  • Do not show hidden validation during broad search.
  • Do not hide failed runs.
  • Do not imply diagnostics are optimization targets.
  • Do not present toy/sample overlays as benchmark evidence.
  • Do not frame the work as improving, reproducing, training, or beating Google DeepMind AlphaFold3.
